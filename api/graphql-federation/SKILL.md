# GraphQL Federation

## Architecture Overview

Apollo Federation v2 composes independently developed **subgraphs** into a unified **supergraph** served by a **router**.

- **Subgraph**: A standalone GraphQL service owning a domain slice. Uses federation directives to declare shared types and cross-service relationships.
- **Supergraph**: The composed schema clients query. Generated by composing all subgraph schemas.
- **Router**: Apollo Router (Rust binary) receives client queries, builds query plans, dispatches subqueries to subgraphs, and assembles responses.

Declare Federation v2 in every subgraph schema:

```graphql
extend schema
  @link(url: "https://specs.apollo.dev/federation/v2.7",
    import: ["@key", "@shareable", "@external", "@requires",
             "@provides", "@override", "@inaccessible", "@tag"])
```

## Federation Directives

### @key — Define entity identity

Mark a type as an entity resolvable across subgraphs. Every subgraph contributing fields must declare `@key`.

```graphql
type Product @key(fields: "upc") { upc: String!; name: String!; price: Int! }

# Multi-field/nested keys
type Review @key(fields: "id author { id }") { id: ID!; author: User!; body: String! }

# Multiple keys for different resolution paths
type User @key(fields: "id") @key(fields: "email") { id: ID!; email: String! }
```

### @shareable — Allow multiple subgraphs to resolve a field

Use on fields or types resolved by multiple subgraphs independently.

```graphql
type Product @key(fields: "upc") {
  upc: String!
  name: String! @shareable       # multiple subgraphs can resolve
  inStock: Boolean!              # only this subgraph resolves
}

# Apply to entire shared value types
type Money @shareable { amount: Int!; currency: String! }
```

### @external — Reference a field defined elsewhere

Declare a field from another subgraph, needed locally for `@requires` or `@provides`.

### @requires — Computed fields needing external data

Tell the router to fetch specified external fields before calling this resolver.

```graphql
type Product @key(fields: "upc") {
  upc: String!
  weight: Int @external
  size: Int @external
  shippingEstimate: Int @requires(fields: "weight size")  # resolver receives both
}

### @provides — Declare additional resolvable fields on return types

Signal that a relationship field can also provide specific nested fields, avoiding an extra subgraph hop.

```graphql
type Review {
  product: Product @provides(fields: "name")
}
extend type Product @key(fields: "upc") {
  upc: String! @external
  name: String! @external
}
```

### @override — Migrate field ownership between subgraphs

Transfer field resolution from one subgraph to another. Supports progressive rollout.

```graphql
type Product @key(fields: "upc") {
  upc: String!
  name: String! @override(from: "legacy-products")
  # Progressive: name: String! @override(from: "legacy", label: "percent(5)")
}

### @inaccessible — Hide fields from the public API

Keep a field for internal composition but exclude from the client-facing API.

```graphql
type Product @key(fields: "upc") { upc: String!; internalCost: Int @inaccessible }
```

## Entity Resolution and Reference Resolvers

Every subgraph that defines fields on an entity must implement a **reference resolver** (`__resolveReference`).

### Node.js / Apollo Server

```typescript
const resolvers = {
  Product: {
    __resolveReference(ref: { upc: string }, context) {
      return context.dataSources.products.getByUpc(ref.upc);
    },
  },
};
```

### Batch reference resolution (critical for performance)

Use DataLoader or batch-capable data sources to avoid N+1 queries:

```typescript
Product: {
  async __resolveReference(refs, context) {
    // refs is an array when batching is enabled
    const upcs = refs.map(r => r.upc);
    const products = await context.dataSources.products.getByUpcs(upcs);
    return upcs.map(upc => products.find(p => p.upc === upc));
  }
}
```

Enable entity batching in subgraph setup to receive arrays.

### Java / DGS Framework

```java
@DgsEntityFetcher(name = "Product")
public Product product(Map<String, Object> values) {
    return productService.findByUpc((String) values.get("upc"));
}
```

### Python / Strawberry

```python
@strawberry.federation.type(keys=["upc"])
class Product:
    upc: str
    name: str

    @classmethod
    def resolve_reference(cls, upc: str) -> "Product":
        return product_repo.get(upc)
```

## Schema Composition and Validation

### Rover CLI commands

```bash
# Compose supergraph locally
rover supergraph compose --config supergraph.yaml --output supergraph.graphql

# Validate a subgraph against the supergraph
rover subgraph check my-graph@prod --name products --schema ./products.graphql

# Publish a subgraph to GraphOS
rover subgraph publish my-graph@prod --name products \
  --schema ./products.graphql --routing-url https://products.internal/graphql

# Introspect a running subgraph
rover subgraph introspect http://localhost:4001/graphql
```

### supergraph.yaml

```yaml
federation_version: =2.7.1
subgraphs:
  products:
    routing_url: http://products:4001/graphql
    schema:
      file: ./schemas/products.graphql
  reviews:
    routing_url: http://reviews:4002/graphql
    schema:
      file: ./schemas/reviews.graphql
  users:
    routing_url: http://users:4003/graphql
    schema:
      subgraph_url: http://users:4003/graphql  # introspect live
```

### Composition errors to watch for

- **Type conflict**: Same field with different types across subgraphs.
- **Missing @key**: Entity referenced but key not declared.
- **@requires referencing non-existent fields**: External field must exist in owning subgraph.
- **Inconsistent nullability**: Align nullable/non-nullable across subgraphs.

See [assets/router.yaml](assets/router.yaml) for a full production-ready configuration.

```yaml
supergraph:
  listen: 0.0.0.0:4000
  path: /graphql
headers:
  all:
    request:
      - propagate:
          named: "Authorization"
traffic_shaping:
  all:
    deduplicate_query: true
    timeout: 30s
telemetry:
  exporters:
    tracing:
      otlp:
        enabled: true
        endpoint: http://jaeger:4317
cors:
  origins:
    - https://app.example.com
```

### Running the router

```bash
# Local development
router --config router.yaml --supergraph supergraph.graphql --dev

# Production with managed federation
APOLLO_KEY=service:my-graph:key APOLLO_GRAPH_REF=my-graph@prod router --config router.yaml
```

## Authentication and Authorization Patterns

### Pattern 1: Gateway-level JWT validation (recommended)

```yaml
authentication:
  router:
    jwt:
      jwks:
        - url: https://auth.example.com/.well-known/jwks.json
      header_name: Authorization
      header_value_prefix: "Bearer "
authorization:
  require_authentication: false
```

### Pattern 2: Propagate claims to subgraphs via headers

```yaml
headers:
  all:
    request:
      - insert:
          name: "X-User-ID"
          value: "{$context.claims.sub}"
```

Subgraphs read user context from these headers—no JWT parsing needed.

### Pattern 3: Schema-level authorization directives

```graphql
type Query {
  publicData: String
  myProfile: User @authenticated
  adminDashboard: Dashboard @requiresScopes(scopes: [["admin"]])
}
```

The router enforces these before executing resolvers.

### Pattern 4: Resolver-level checks for fine-grained ABAC

```typescript
order: (_, { id }, context) => {
  const order = orderService.get(id);
  if (order.userId !== context.userId) throw new ForbiddenError();
  return order;
},
```

## Performance Optimization

### Query planning

- Apollo Router caches query plans. First request is slower; subsequent identical queries reuse the plan.
- Reduce plan complexity: avoid deeply nested cross-subgraph entity chains.
- Use `@provides` to short-circuit hops when a subgraph already has the data.

### Subgraph batching

Enable in router config to batch multiple entity fetches into a single request per subgraph:

```yaml
traffic_shaping:
  all:
    experimental_entity_caching: true
  subgraphs:
    products:
      experimental_batching:
        enabled: true
```

### DataLoader in subgraphs

Always use DataLoader for entity resolution to prevent N+1:

```typescript
const productLoader = new DataLoader(async (upcs: string[]) => {
  const products = await db.products.findMany({ where: { upc: { in: upcs } } });
  return upcs.map(upc => products.find(p => p.upc === upc));
});
```

### Deduplication

Router deduplicates identical in-flight requests to the same subgraph. Enable `deduplicate_query: true` globally or per subgraph.

### Response caching

Use `@cacheControl` to set per-field TTLs:

```graphql
type Product @key(fields: "upc") @cacheControl(maxAge: 300) {
  upc: String!
  name: String! @cacheControl(maxAge: 3600)
  inventory: Int! @cacheControl(maxAge: 30)
}
```

## Subgraph Design Patterns

### Domain-driven subgraph boundaries

Align subgraphs to bounded contexts. Each team owns one or more subgraphs:

| Domain          | Subgraph     | Entities owned        |
|-----------------|------------- |-----------------------|
| Product catalog | products     | Product, Category     |
| User management | users        | User, Account         |
| Orders          | orders       | Order, OrderItem      |
| Reviews         | reviews      | Review, Rating        |
| Inventory       | inventory    | StockLevel, Warehouse |

### Entity ownership rules

- **One owner per entity**: A single subgraph defines the canonical entity with `@key` and core fields
- **Contributors extend**: Other subgraphs add fields by declaring the entity with `@key` and their fields only
- **Never duplicate business logic**: Use `@shareable` or extract to a shared subgraph

See [subgraph-patterns.md](references/subgraph-patterns.md) for deep dives on entity design, @key variations, interface entities, and more.

## Managed Federation with GraphOS

### Setup

1. Create a graph in Apollo Studio / GraphOS.
2. Publish each subgraph schema with `rover subgraph publish`.
3. GraphOS composes the supergraph, runs validation, and stores the schema.
4. Router polls GraphOS for schema updates via `APOLLO_KEY` and `APOLLO_GRAPH_REF`.

### CI/CD integration

```yaml
# .github/workflows/schema-check.yml
- name: Check subgraph schema
  run: rover subgraph check $APOLLO_GRAPH_REF --name $SUBGRAPH_NAME --schema ./schema.graphql
  env:
    APOLLO_KEY: ${{ secrets.APOLLO_KEY }}
```

See [schema-check.sh](scripts/schema-check.sh) for a comprehensive local + remote validation script.

### Benefits

- Prevents breaking changes from reaching production
- Tracks field usage metrics; provides schema history and audit log
- Enables contract variants for different API consumers

## Migrating from Monolithic GraphQL

1. Deploy router in front of monolith as the sole subgraph (zero behavior change)
2. Annotate entities with `@key`; add `__resolveReference` stubs
3. Extract one domain at a time using `@override` for gradual field migration
4. Validate with `rover subgraph check` on every change
5. Repeat for each bounded context

See [migration-guide.md](references/migration-guide.md) for detailed patterns including schema stitching migration, progressive `@override`, data migration, testing, and rollback strategies.

## Alternatives Comparison

| Approach            | Best for                               | Trade-offs                                    |
|---------------------|----------------------------------------|-----------------------------------------------|
| Apollo Federation   | Microservices, large orgs, team autonomy | Steeper learning curve, operational complexity |
| Schema Stitching    | Quick aggregation of existing GraphQL APIs | Gateway bottleneck, manual type merging       |
| GraphQL Mesh        | Unifying REST + gRPC + GraphQL sources | Performance overhead, less schema governance   |
| WunderGraph Cosmo   | Vendor-neutral federation alternative  | Smaller ecosystem, newer                      |
| GraphQL Hive        | Open-source schema registry + gateway  | Requires self-hosting or Hive Cloud           |

Choose **Apollo Federation** for greenfield distributed GraphQL. Use **Mesh** when integrating heterogeneous non-GraphQL APIs.

## Common Pitfalls

- **Missing __resolveReference**: Composition succeeds but runtime fails.
- **N+1 in entity resolvers**: Always use DataLoader or batch fetching.
- **Over-sharing with @shareable**: Only share fields with identical semantics across subgraphs.
- **Circular entity dependencies**: Keep dependency graphs acyclic.
- **Large monolith subgraph**: Decompose aggressively; avoid one subgraph owning most types.
- **Ignoring composition errors**: Treat as build failures in CI.
- **Not using managed federation in production**: Local composition is for dev only.

See [troubleshooting.md](references/troubleshooting.md) for diagnosis and fixes.
